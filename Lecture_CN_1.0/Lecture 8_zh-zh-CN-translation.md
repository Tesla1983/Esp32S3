## 蓝牙

### 介绍

到目前为止，我们已经探讨了有线通信协议，如 **UART、I²C 和 SPI**。这些协议是嵌入式系统设计师工具包中的关键工具。它们使我们能够扩展微控制器的功能、添加外部传感器、与显示屏交互，并通过共享总线连接多个微控制器。

然而，有线通信带来了显著的局限性：

*   **缺乏移动性：** 设备必须通过物理方式连接在一起。
    
*   **短距离限制：** 像 I²C 和 SPI 这类协议专为短距离通信设计（通常在同一块电路板上）。若通过长距离线缆延伸，则会引入噪声和信号衰减问题。
    
*   **可扩展性与成本：** 在整栋建筑内为数十个环境传感器铺设物理线缆既昂贵又不切实际。
    

无线通信协议通过无线电波在空中传输和接收数据来解决这些问题。这种方式无需物理线缆，实现了真正的移动性、更远的传输距离，以及复杂传感器网络的便捷部署。嵌入式领域最常见的两种无线协议是：适用于高带宽互联网连接的 Wi-Fi，以及适合低功耗设备间通信的蓝牙。

### 蓝牙协议

蓝牙是一种短距离无线通信标准，工作在 2.4 GHz ISM（工业、科学和医疗）无线电频段。它旨在通过低功耗无线电通信，在短距离内实现设备间的数据交换。

蓝牙并非传输连续信号，而是将数据分割成称为数据包的小单元。这些数据包通过一种称为跳频的技术在空中传输，即无线电在 2.4 GHz 频段内的不同频率之间快速切换。这能减少来自共享同一频段的其他设备（如 Wi-Fi 路由器和微波炉）的干扰。

2.4 GHz 频段被划分为多个信道。经典蓝牙使用 79 个信道，每个信道间隔 1 MHz；而低功耗蓝牙（BLE）使用 40 个信道，每个信道间隔 2 MHz。在通信过程中，设备遵循共同的跳频序列，并按该序列在不同信道上传输每个数据包。

为了交换数据，蓝牙通常依赖于中心/外围设备架构：

*   **外围设备（服务器）：** 一种广播自身存在并持有数据的设备。
*   **中心设备（客户端）：** 一种扫描外围设备、连接它们并请求或发送数据的设备。

### 蓝牙网络架构

#### 微微网

当设备通过蓝牙连接时，它们会形成一个临时的微型网络，称为**微微网** 。在微微网中，设备承担特定的角色：

1.  **主设备：** 发起连接并决定跳频序列和时序的设备（例如，你的智能手机）。
2.  **从设备：** 遵循主设备规则并与其时钟同步的设备（例如，你的无线耳机和智能手表）。

一个微微网可同时连接一个主设备和最多七个活跃的从设备。主设备与从设备通信，但从设备之间不直接通信；所有信息都通过主设备进行路由。

![](./attachments/network1.png)

#### 散射网

当多个微微网通过参与多个微微网的设备连接时，便形成了散射网，从而让更大的网络能够无缝运行。

![](./attachments/network2.png)

在散射网中，一个设备可以在一个微微网中作为从设备，同时在另一个微微网中充当主设备，从而形成互联的无线节点网络。

#### 连接过程

两个设备要建立网络并共享数据，需要经历特定的连接过程：

*   **查询：** 如果某个设备想要寻找其他设备，它会发送查询请求。而希望被发现的设备必须处于"可发现模式"，即主动监听并响应这些请求。
*   **寻呼：** 设备相互发现后，便会建立连接。这包括同步它们的时钟并商定跳频模式。
*   **配对（安全）：** 为确保邻居不会意外连接到你的扬声器，设备通常需要配对过程。它们会生成并交换一个安全的数字密钥（通常通过输入 PIN 码或确认提示来验证）。配对后，设备会记住此密钥，未来可使用加密信号自动连接，从而保护数据不被窃听。

### 蓝牙 Variant

随着蓝牙技术的发展，为解决一个关键限制——功耗问题，一种重要的新变体应运而生。

#### 蓝牙 Classic

最初的蓝牙，通常称为经典蓝牙或 BR/EDR（基本速率/增强数据速率），专为连续数据流传输而设计。它提供高达 3 Mbps 的数据速率，非常适合音频流（耳机、扬声器）或快速文件传输等应用。然而，它的功耗相对较高，这对于需要靠纽扣电池运行数月的小型电池供电设备（如健身手环或智能传感器）来说是个问题。

#### 蓝牙低功耗（BLE）

蓝牙低功耗（BLE）于 2010 年随蓝牙 4.0 版本推出，其设计初衷便将能效置于首位。BLE 通过以下设计选择实现了显著更低的功耗：

*   **短时突发传输** ：BLE 设备并非维持持续连接，而是大部分时间处于休眠状态，仅在发送或接收小数据包时短暂唤醒。
*   **更少的信道** ：BLE 仅使用 40 个信道（每个带宽 2 MHz），其中 3 个为专用广播信道，用于设备发现。
*   **更简单的连接过程** ：BLE 连接的建立和终止速度更快。
*   **优化的数据包结构** ：BLE 数据包更小、更简单。

一个 BLE 设备可以依靠小型纽扣电池运行数月甚至数年，因此非常适合用于：

*   健身追踪器和智能手表
*   医疗传感器（心率监测器、血糖仪）
*   智能家居设备（门锁、灯具、恒温器）
*   信标技术（商店内的定位服务）
*   工业物联网传感器

### 蓝牙协议栈

蓝牙并非单一协议，而是一组按层组织的协议栈。每一层负责通信的特定部分，从物理无线电传输一直到用户交互的应用程序。

![](./attachments/bluetooth_stack.png)

蓝牙协议栈主要分为两大部分：主机栈和控制器栈

### 控制器栈

控制器栈管理硬件级操作和底层链路控制，包含以下层级：

#### 物理层（射频）

协议栈最底层为射频层，即实际硬件部分：负责发射与接收无线电波的天线，以及执行调制解调功能的电路。该层通过 GFSK 调制和 FHSS 跳频技术，实现原始比特数据的空中传输。

#### 基带层

物理无线电之上是基带层。该层负责管理比特分组与传输的时序和结构，具体处理以下内容：

*   **数据包形成** ：数据被组织成数据包（小型数据单元），其头部包含寻址和控制信息。
*   **错误检测** ：采用循环冗余校验（CRC）等技术来检测传输错误。
*   **同步机制** ：利用主设备的时钟确保微微网中所有设备保持同步。
*   **链路类型** ：蓝牙支持两种链路：
    *   **SCO（同步面向连接）**：用于实时音频传输，允许少量数据丢失但必须保证精确时序，适用于通话场景。
    *   **ACL（异步无连接）**：用于常规数据传输，要求数据完整无损，适用于文件传输、传感器等场景。

#### LMP：链路管理协议

链路管理器协议（LMP）负责建立和管理蓝牙链路。它处理：

*   配对与认证：验证双方设备身份的真实性。
*   加密：对数据进行加扰，防止未授权监听者读取。
*   电源管理：协商低功耗休眠状态以节省电量。
*   角色切换：设备在必要时可请求从从设备切换为主设备。

### 主机控制器接口

HCI（主机控制器接口）是蓝牙硬件（射频 + 基带 + LMP）与主机处理器（通常是微控制器或 CPU）上运行的软件之间的标准化边界。它定义了一组主机可向控制器发送的命令，以及控制器返回的事件。

这种分离意味着蓝牙射频可以作为独立芯片，通过 UART 或 USB 等简单接口与主处理器通信。

### 主机协议栈

#### L2CAP：逻辑链路控制与适配协议

L2CAP 是蓝牙协议栈中的一层，负责帮助不同应用程序共享同一条蓝牙连接。它作为应用程序与底层蓝牙系统之间的中间层发挥作用。

其主要功能是多路复用，即允许多个数据流同时使用同一连接而不会相互混淆。每个应用程序都拥有独立的逻辑通道。

L2CAP 还负责数据包的分割与重组。如果数据过大，它会在发送前将其拆分为更小的片段。在接收端，它将这些片段重新组合，确保原始数据正确还原。

它还支持服务质量（QoS），可根据应用需求决定数据的处理方式。例如，音频传输需要快速送达，而文件传输则更注重准确性。

#### 蓝牙上层协议

在 L2CAP 层之上，蓝牙使用多种上层协议来提供不同的通信服务，例如串行通信、文件传输、互联网接入、音频流和电话服务。这些协议使蓝牙设备能够支持多种类型的应用和服务。

**RFCOMM** 是一种蓝牙协议，功能类似于简单的串行线缆替代方案。它模拟传统的串行（RS-232）通信，因此原本为有线串口设计的旧应用程序可以轻松通过蓝牙运行。该协议基于 L2CAP 协议，广泛应用于微控制器项目中的蓝牙模块、GPS 设备以及简易无线传感器等设备。同时，它遵循 ETSI TS 07.10 标准，该标准定义了串行通信的处理方式。

**OBEX（对象交换协议）** 用于在设备间传输文件、图片和联系人等对象，常见于文件共享和联系人同步功能中。

**WAP（无线应用协议）** 曾用于早期移动设备通过蓝牙及其他无线网络访问基础互联网服务，但目前基本已被淘汰。

**TCS（电话控制协议）** 用于管理语音通信服务，例如建立和结束通话，主要应用于无绳电话系统和语音网关。

此级别的其他协议包括：

*   **BNEP**（蓝牙网络封装协议）：用于通过蓝牙进行网络连接。
*   **AVDTP**（音频/视频分发传输协议）：用于音频流传输。
*   **AVCTP**：用于音频/视频远程控制。

### 配置文件层

协议栈的顶层是配置文件。配置文件是一种规范，明确定义了设备为执行特定功能必须支持的协议和特性。配置文件确保互操作性：如果两个设备都实现了相同的配置文件，它们就能保证协同工作。

一些重要的蓝牙配置文件包括：

| 配置文件 | 缩写 | 用途 |
| --- | --- | --- |
| 串行端口配置文件 | SPP | 无线串行连接 |
| 免提配置文件 | HFP | 通过车载音响或耳机进行通话 |
| 高级音频分发配置文件 | A2DP | 高质量立体声音频流 |
| 音频/视频远程控制配置文件 | AVRCP | 媒体播放控制（播放、暂停、跳过） |
| 人机接口设备配置文件 | HID | 键盘、鼠标、游戏控制器 |
| 文件传输配置文件 | FTP | 在设备之间发送文件 |
| 健康设备配置文件 | HDP | 医疗传感器数据（血压、血糖监测仪） |

### BLE 架构

低功耗蓝牙（BLE）在经典蓝牙的基础上引入了多项改进和调整，特别是在主机层，通过定义自身轻量级且节能的协议结构来实现。BLE 专为物联网设备（如传感器、智能开关、可穿戴设备和医疗设备）中的低功耗无线通信而设计。

BLE 架构分为多个层级，每个层级负责特定的通信功能。主机层包含以下重要协议：

*   **SMP（安全管理协议）** 用于安全与配对
*   **GAP（通用访问规范）** 用于设备发现和连接管理
*   **ATT/GATT（属性协议/通用属性规范）** 用于数据组织和交换
*   **服务发现机制**用于识别可用的服务和特征

### 使用 Esp32s3 实现蓝牙功能

ESP32-S3 仅支持低功耗蓝牙（BLE）。其通用 BLE 实现与我们之前讨论的内容类似，但 ESP-IDF 框架提供了两个内置主机栈（Bluedroid 和 NimBLE）以简化开发。此外，ESP-IDF 还提供了多个可直接使用的 BLE 配置文件和服务（如 BluFi、HID 和 ESP-BLE-MESH），可基于这些协议栈使用，从而加速应用程序开发。

![](./attachments/esp32ble_ar.png)

### 主机栈

#### ESP-Bluedroid 与 ESP-NimBLE

ESP-Bluedroid 是 ESP-IDF 提供的默认蓝牙协议栈。它基于 Android Bluedroid 协议栈开发，在兼容的 ESP 设备上同时支持经典蓝牙和低功耗蓝牙（BLE）。对于仅支持 BLE 的 ESP32-S3，ESP-Bluedroid 提供了完整的 BLE 主机实现，支持 GAP、GATT、安全功能及多种 BLE 配置文件。该协议栈功能丰富，与 ESP-IDF 生态系统深度集成，适用于复杂的 BLE 应用场景。

#### ESP-NimBLE

ESP-NimBLE 是 ESP-IDF 中提供的一款轻量级、资源高效的 BLE 主机协议栈。它基于 Apache NimBLE 项目，相比 Bluedroid，针对低内存占用和更优性能进行了优化。ESP-NimBLE 支持 GAP、GATT、广播、扫描和安全连接等标准 BLE 功能，同时消耗更少的系统资源。由于其占用空间小且效率高，在内存和功耗至关重要的嵌入式应用中，它通常是更受青睐的选择。

#### 配置文件

在主机协议栈之上，是乐鑫实现的配置文件以及一些通用配置文件。根据您的配置，这些配置文件可以在 ESP-Bluedroid 或 ESP-NimBLE 上运行。

#### ESP-BLE-MESH 协议栈

ESP-BLE-MESH 是 ESP-IDF 提供的一种低功耗蓝牙网状网络框架。它支持多个 BLE 设备以多对多拓扑结构相互通信，使消息能够在大规模节点网络中中继传输。这使得 ESP-BLE-MESH 适用于智能家居、工业自动化、传感器网络和智能照明系统等应用场景。该框架基于官方蓝牙 Mesh 标准，支持设备配置、安全通信、消息中继和节点管理。通过使用 ESP-BLE-MESH，开发者可以构建可扩展且可靠的物联网网络，其通信范围远超标准点对点 BLE 连接。

#### BluFi 协议栈

BluFi 是乐鑫科技开发的一种专有低功耗蓝牙（BLE）配置文件，旨在简化 ESP 设备的 Wi-Fi 配置流程。它允许智能手机或其他支持 BLE 的设备通过 BLE 连接，安全地将 Wi-Fi 凭证（如 SSID 和密码）传输至 ESP 设备。接收凭证后，ESP 设备可自动连接至目标 Wi-Fi 网络。BluFi 常用于物联网产品中，提供简便易用的配置过程，无需临时接入点或手动配置。ESP-IDF 内置了对 BluFi 配置文件的支持，使开发者能够轻松集成。
